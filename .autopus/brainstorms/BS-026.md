## 원본 아이디어
- What: SPEC review REVISE 루프의 수렴 보장을 위한 2-Phase Scoped Review 패턴 도입
- Why: 리뷰어가 매 라운드마다 새 코드 경로를 탐색하면서 finding이 줄지 않고 늘어나는 비수렴 문제
- Who: autopus-adk 파이프라인 사용자 (개발자)
- When: 현재 파이프라인 품질 게이트 신뢰성 문제 — 즉시 필요

## 근본 원인 분석

1. **Stateless Review**: `BuildReviewPrompt`(prompt.go:11)에 mode 파라미터 없음, 매번 동일한 open-ended 리뷰 지시
2. **Finding 미추적**: `ReviewFinding`(types.go:69-74)에 ID/Status 없음, `spec_review.go`에서 blind append
3. **Discovery/Verification 미분리**: 첫 리뷰와 재리뷰가 동일한 탐색적 모드

## 프로바이더별 발산 결과

### claude (토론자 A)
- Finding Ledger + Canonical Form: finding_id + category + scope_ref + status + first_seen_rev
- ScopeManifest + coverage_receipt.json: "무엇을 봤는지" 저장
- Out-of-scope candidate 큐: Phase B에서 신규 finding 완전 금지 대신 보류 큐
- FindingDedup: ScopeRef + Category 기준 deterministic 통합 (LLM 호출 없음)
- ParseVerdict 파서 레벨 강제: priorFindings ID 목록에 없는 finding 자동 태깅

### codex (토론자 B)
- Mode-aware builder: discover 모드(전수 스캔) / verify 모드(체크리스트 + diff)
- 정적 분석 Integration: lint/staticcheck를 Phase A deterministic lane으로 통합
- 멀티프로바이더 Phase A 집중 배치: finding마다 consensus 태그
- UX 전환: "체크리스트가 얼마나 닫혔는가" 상태표
- Circuit breaker: finding 수 증가 시 자동 중단
