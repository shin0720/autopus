# BS-020: Worker ADK 비사용 유저를 위한 UX 설계 — 플랫폼 전환 퍼널

**Created**: 2026-04-02
**Strategy**: debate
**Providers**: claude, codex, gemini
**Status**: active

## 원본 아이디어
- What: Worker ADK를 안 쓰는 오픈소스 유저를 위한 UX 설계 — Help 카테고리 분리, 초기 setup 분기, autopus.yaml worker 설정, 자연스러운 업셀 힌트
- Why: ADK는 오픈소스 → 플랫폼 전환 퍼널. 현재 Worker가 Core와 구분 없이 노출되어 인지 부하 + 플랫폼 발견 동선 부재
- Who: ADK를 개발 도구(plan/go/sync)로만 사용하는 오픈소스 개발자 + 잠재적 플랫폼 전환 유저
- When: Worker ADK(SPEC-ADKW-001) 구현 안정화 후, UX 레이어 개선 (다음 마이너 릴리즈)

## 핵심 가정 및 리스크

| 축 | 가정 | Risk Level |
|---|---|---|
| **Value** | 오픈소스 유저가 Worker/플랫폼으로 전환하고 싶어한다 | HIGH |
| **Usability** | Help 분리와 힌트만으로 발견 동선이 충분하다 | MEDIUM |
| **Viability** | 업셀 힌트가 오픈소스 커뮤니티에서 반감을 사지 않는다 | HIGH |

## 합의 영역 (2+ 프로바이더 공통)

1. **초기 설정 분기**: Developer/Fullstack 모드 선택으로 첫 경험 커스터마이징
2. **Help 계층화**: 역할/use-case 기반 Help 동적 그룹핑
3. **비침습적 가이드**: 행동 기반 트리거로 자연스러운 기능 발견
4. **커뮤니티 신뢰 보호**: 상업적 브랜딩 최소화, 기여 CTA 우선
5. **초기 복잡성 제거**: Auto-bootstrap으로 빈 프로젝트 파라리시스 해소

## 교차 리스크 (공통 우려)

- 오픈소스 커뮤니티 "상업 미끼" 반감
- 강제성/선택지 부족으로 인한 이탈
- 초기 설정 복잡성
- 플랫폼 전환 타이밍의 모호함

## 고유 인사이트

- **빌드 태그 기반 선택적 Worker 포함** (Go 빌드 태그로 바이너리 분리)
- **"한 줄, 한 번" 업셀 원칙**: 최대 2회(첫 성공 + 반복 감지)로 제한
- **autonomy.level 3단계**: manual/assisted/autonomous — Worker 개념 대신 자율화 레벨로 프레이밍
- **커뮤니티 기여 CTA**: 플랫폼 광고 대신 "이 결과를 커뮤니티에 공유하세요" 우선
- **Homebrew Core/Full 분리 배포 패턴 적용**
